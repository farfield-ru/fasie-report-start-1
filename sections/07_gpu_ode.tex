\chapter{ЧИСЛЕННОЕ ИНТЕГРИРОВАНИЕ ДИФФЕРЕНЦИАЛЬНЫХ УРАВНЕНИЙ С ПОМОЩЬЮ GPU}
\label{cha:gpu}

\section{Постановка задачи}

Центральная идея проекта --- перенос расчёта физической динамики на графический
ускоритель. Графические процессоры содержат тысячи вычислительных ядер и при
правильной организации данных способны существенно ускорять массовые однотипные
вычисления, к которым относится интегрирование уравнений движения
\eqref{eq:newton-euler} множества тел. Настоящая работа непосредственно
продолжает предыдущую (раздел~\ref{cha:cpu}): благодаря принятому решению о
единой кросс-платформенной реализации задача состояла не в написании отдельной
GPU-версии, а в \emph{оптимизации того же исходного кода} для эффективного
исполнения на графическом ускорителе --- организации памяти, минимизации
синхронизаций, распараллеливании всех стадий конвейера и обеспечении
детерминизма параллельных вычислений. Работа выполнена в репозитории
\texttt{dyphur}.

\section{Выбор технологии гетерогенных вычислений}

Для аппаратного ускорения рассматривались четыре технологии. CUDA обеспечивает
наилучшую производительность на оборудовании NVIDIA и богатую экосистему
инструментов, однако жёстко привязывает проект к одному производителю и
исключает ускорители AMD. OpenCL --- открытый стандарт с широкой поддержкой
устройств, но его активное развитие фактически прекратилось, а синтаксис
OpenCL~C далёк от современного C++. Vulkan Compute~\cite{vulkan}
кроссплатформенен и имеет низкие накладные расходы, но требует ручного
управления промежуточным представлением SPIR-V, дескрипторами и барьерами
памяти --- значительная сложность без прямых преимуществ для вычислительных
задач физики.

Выбран \textbf{SYCL~2020}~\cite{sycl2020} --- открытый стандарт ISO/Khronos
гетерогенного программирования на C++ --- в реализации
\textbf{AdaptiveCpp}~\cite{alpay2020adaptivecpp} (ранее hipSYCL). Преимущества
в контексте проекта:
\begin{enumerate}
  \item \emph{единый исходный код}: ядро на C++20 компилируется для CUDA, HIP
        и центрального процессора без изменений, бэкенд задаётся флагами
        сборки CMake (таблица~\ref{tab:backends});
  \item \emph{современный C++}: лямбда-ядра, шаблоны, \texttt{constexpr}
        доступны в коде устройства без отдельного диалекта;
  \item \emph{единая модель памяти} (Unified Shared Memory, USM): память
        выделяется непосредственно на устройстве, обмен данными выполняется
        явными операциями копирования;
  \item \emph{детерминизм на уровне примитивов}: поддержка упорядоченных
        редукций позволяет гарантировать побитовую воспроизводимость.
\end{enumerate}

\section{Переносимый вычислительный слой}

Прямые вызовы SYCL инкапсулированы в модуле \texttt{compute}
(рисунок~\ref{fig:compute-arch}); физическое ядро обращается к ускорителю
только через интерфейсы этого слоя:
\begin{itemize}
  \item \texttt{Device} --- вычислительное устройство; выбор задаётся методами
        \texttt{default\_gpu()} и \texttt{default\_cpu()};
  \item \texttt{Stream} --- очередь команд в режиме \emph{in-order}: все
        операции выполняются строго в порядке постановки;
  \item \texttt{Buffer<T>} --- владеющий буфер в USM-памяти устройства с
        асинхронными операциями \texttt{upload} (ЦП$\to$ГП) и
        \texttt{download} (ГП$\to$ЦП);
  \item \texttt{parallel\_for(s, n, f)} --- запуск вычислительного ядра:
        лямбда \texttt{f(i)} вызывается параллельно для каждого
        $i \in [0, n)$;
  \item \texttt{reduce\_sum}, \texttt{sort\_by\_key} --- детерминированные
        коллективные примитивы (редукция по дереву с фиксированным порядком
        операций, сортировка по ключу).
\end{itemize}

В код устройства не проникают контейнеры STL, исключения и виртуальные
функции; контроль обеспечивается отдельной проверкой зависимостей на этапе

\section{Оптимизация конвейера для графического процессора}

\textbf{Жизненный цикл данных.} Данные сцены копируются на устройство один раз
при инициализации; на каждом шаге вычислительные ядра модифицируют их на
месте, без переноса на центральный процессор. Единственная синхронизация
ЦП$\leftrightarrow$ГП за шаг --- чтение счётчика кандидатных пар после широкой
фазы (одно 4-байтное чтение). Порядок операций одного шага показан на

\textbf{Интегратор.} Интегрирование уравнений движения распараллелено по
телам: каждому телу сопоставляется отдельный рабочий элемент (work-item),
выполняющий симплектический шаг \eqref{eq:symplectic} и обновление кватерниона
\eqref{eq:quat-integrate}. SoA-раскладка обеспечивает слитный доступ к
глобальной памяти: соседние рабочие элементы обращаются к соседним ячейкам
массивов. Ключевой фрагмент реализации приведён в листинге~\ref{lst:gpu}.

\begin{lstlisting}[language=C++,caption={Ядро интегрирования (\texttt{core/src/integrator.cpp}, фрагмент)},label=lst:gpu]
void integrate(Stream& s, BodyView bodies, const IntegratorParams& params) {
    const float sub_dt = params.dt / float(params.n_substeps);
    for (int sub = 0; sub < params.n_substeps; ++sub) {
        parallel_for(s, bodies.n, [=](size_t i) {
            // Статические, кинематические и «спящие» тела пропускаем.
            if (bodies.inv_mass[i] == 0.f) return;
            if (bodies.flags[i] & BodyFlag::Sleeping) return;
            // Симплектический Эйлер: сначала скорость, затем позиция.
            float vx = bodies.vel_x[i] + gx * sub_dt;   // аналогично vy, vz
            bodies.vel_x[i] = vx;
            bodies.pos_x[i] += vx * sub_dt;
            // Кватернион: q += 0.5 * sub_dt * (0, omega) ⊗ q ...
            // ... затем перенормировка для компенсации дрейфа:
            const float inv_n = sycl::rsqrt(qw*qw + qx*qx + qy*qy + qz*qz);
            bodies.rot_w[i] = qw * inv_n;               // аналогично x, y, z
        });
    }
}
\end{lstlisting}

\textbf{Широкая фаза.} Все четыре прохода LBVH (раздел~\ref{cha:cpu})
выполняются на устройстве: коды Мортона и узлы дерева Карраса вычисляются по
одному потоку на элемент. Обновление AABB организовано по уровням
(level-synchronized refit): каждый раунд слияния --- отдельная постановка ядра
в очередь, благодаря чему родительский узел читает только значения, записанные
предшествующим ядром. Это устраняет состояние гонки данных и обеспечивает
воспроизводимость на всех бэкендах. AABB узлов хранятся в формате fp16, что
вдвое снижает потребление памяти и нагрузку на её полосу пропускания.

\textbf{Узкая фаза.} Каждая кандидатная пара обрабатывается одним потоком;
контакты накапливаются в промежуточный буфер и затем сортируются по
каноническому ключу --- порядок контактов воспроизводит последовательный
обход.

\textbf{Решатель.} Итерации XPBD распараллелены методом \emph{раскраски графа
ограничений}: ограничения разбиваются на цвета так, чтобы внутри одного цвета
никакие два ограничения не разделяли динамическое тело. Весь цвет выполняется
одним вычислительным ядром без атомарных операций; цвета обрабатываются строго
по порядку, что сохраняет связность схемы Гаусса--Зейделя между цветами.
Раскраска вычисляется жадным однопроходным алгоритмом в начале каждого шага;
статические тела в граф не включаются.

\section{Детерминизм на GPU}

Параллельные вычисления с плавающей точкой не ассоциативны: порядок
суммирования влияет на результат. Побитовая воспроизводимость обеспечивается
тремя мерами:
\begin{enumerate}
  \item \emph{детерминированная сортировка пар}: кандидатные пары упорядочиваются
        по каноническому ключу до узкой фазы, что фиксирует порядок контактов
        независимо от порядка обхода BVH;
  \item \emph{детерминированные коллективные примитивы}: редукции выполняются по
        дереву с фиксированным порядком операций вместо встроенных атомарных
        сложений, результат которых зависит от порядка прибытия нитей;
  \item \emph{фиксированное число итераций решателя} и фиксированный порядок
        цветов раскраски.
\end{enumerate}
В результате на фиксированной паре <<сборка --- оборудование>> симуляция даёт
побитово идентичный результат при каждом запуске. Хеши состояния для бэкендов
CUDA и OpenMP различаются между собой --- это ожидаемо и обусловлено
неассоциативностью арифметики с плавающей точкой и режимом FTZ на графическом
процессоре, а не ошибкой реализации (см. раздел~\ref{cha:test}).

\section{Результаты}

Реализован и оптимизирован решатель (солвер) дифференциальных уравнений
движения на графическом процессоре поверх переносимого вычислительного слоя
SYCL/AdaptiveCpp с бэкендами CUDA и HIP. Все стадии конвейера --- интегратор,
широкая и узкая фазы, решатель ограничений --- исполняются на устройстве с
единственной синхронизацией с центральным процессором за шаг. Обеспечен
побитовый детерминизм результата. Достигнута симуляция в реальном времени для
сцен до 8192 тел; начиная примерно с 1000 тел GPU-исполнение опережает
CPU-исполнение. Принятое решение о единой кросс-платформенной реализации
позволило выполнить настоящую работу как оптимизацию уже отлаженного кода
(раздел~\ref{cha:cpu}), существенно сократив трудозатраты и исключив
расхождение версий.
